# Security Policy — AERVINEX

We take the security of AERVINEX seriously. AERVINEX is a wearable health
companion handling medical-adjacent biometric data; vulnerabilities can have
real-world impact on user privacy and safety. This policy describes how to
report security issues responsibly and what to expect from us in return.

## Supported Versions

Only the latest production deployment of `aervinex.web.app` and the `main`
branch of this repository are supported.

| Version | Supported          |
|---------|--------------------|
| main / current production | Yes |
| older tags / forks        | No  |

## Reporting a Vulnerability

**Please do NOT open a public GitHub issue for security reports.**

### Preferred channel
Email **security@aervinex.id** (fallback: `security@aervi.id`).

Include:
1. Affected component / URL (e.g. `aervinex.web.app/profile.html`).
2. Vulnerability type (XSS, IDOR, auth bypass, dependency CVE, etc.).
3. Reproduction steps (numbered, ideally with a minimal PoC).
4. Impact assessment — what an attacker could do.
5. Suggested fix (optional but appreciated).
6. Your name / handle if you want credit in the Hall of Fame.

### Encrypted disclosure
For sensitive reports, encrypt with our PGP key. Fingerprint and full key block
are at <https://aervinex.web.app/security.html#pgp>. If the key has not been
published yet, request it at `security@aervinex.id` and we will reply within
24 hours.

## What to Expect

| Stage | SLA |
|-------|-----|
| Acknowledgement of receipt | **72 hours** |
| Initial triage + severity decision | **7 days** |
| Status updates | every **7 days** until resolution |
| Fix for **Critical** (auth bypass, RCE, mass data leak) | **≤ 14 days** |
| Fix for **High** | **≤ 30 days** |
| Fix for **Medium / Low** | **≤ 90 days** |
| Public disclosure | After fix is deployed, or **90 days** after report (whichever first) |

The 90-day disclosure timeline aligns with Project Zero and standard
responsible-disclosure practice. We will request an extension in writing if a
fix legitimately requires more time.

## Scope

### In scope
- `aervinex.web.app`, `aervinex.firebaseapp.com`, and any future custom domain
  (e.g. `app.aervinex.id`).
- Cloud Functions deployed from this repository.
- Firestore security rules (`firestore.rules`).
- Source code in this repository.

### Out of scope
- Denial-of-service (DoS, DDoS, brute force at scale).
- Social engineering of AERVINEX staff or users.
- Physical attacks against user devices.
- Vulnerabilities in third-party services we depend on
  (report to Google / Firebase / Open-Meteo / IQAir / BMKG directly).
- Self-XSS that requires the victim to paste attacker code into DevTools.
- Missing security headers without a demonstrable exploit chain.
- Reports generated by automated scanners without manual verification.
- Outdated browsers (IE 11, Safari &lt; 14, etc.).
- Best-practice suggestions without a concrete vulnerability.

## Safe Harbor

We will not take legal action or contact law enforcement against researchers
who:
- Make a good-faith effort to follow this policy.
- Avoid privacy violations of other users (use your own test account).
- Avoid destructive testing (no data deletion, no DoS).
- Do not publicly disclose before we agree on a coordinated date.
- Do not retain, share, or use accessed data beyond what is needed to
  demonstrate the vulnerability.

If you accidentally access data belonging to another user during testing,
stop immediately and tell us. We will not penalize accidental access reported
in good faith.

## Hall of Fame

Researchers who report a verified vulnerability — and consent to public credit
— will be listed at <https://aervinex.web.app/security.html#hall-of-fame>.
We offer:
- Public credit (name, handle, link of your choice).
- A signed thank-you letter from the team.
- Swag where logistics allow.

We do not currently run a paid bug bounty program. This may change as the
project matures.

## How We Handle Reports Internally

1. **Triage** — security lead validates and assigns severity (CVSS 3.1).
2. **Containment** — for critical issues, temporary mitigation
   (rule tightening, feature flag flip) within 24 hours.
3. **Fix** — patch developed on a private branch; reviewed by &ge; 1 other
   engineer.
4. **Deploy** — fix shipped to production. Reporter notified.
5. **Disclosure** — public advisory posted in `docs/SECURITY-ADVISORIES.md`
   and as a GitHub Security Advisory.
6. **Post-mortem** — for Critical/High, internal post-mortem within 14 days.

Keamanan

Versi 1.0 · 31 Mei 2026

Halaman ini mendokumentasikan postur keamanan teknis AERVINEX: bagaimana data Anda dilindungi, di mana di-host, siapa subprocessor kami, dan bagaimana melaporkan kerentanan.

1. Enkripsi

1.1 Data at Rest

  • Cloud Firestore — semua dokumen di-enkripsi otomatis dengan AES-256 (Google-managed keys). Ini bawaan Google Cloud Platform.
  • Cloud Storage (jika digunakan untuk export PDF) — AES-256 server-side encryption.
  • Backups — Point-in-Time Recovery (PITR) Firestore juga ter-enkripsi sama.

1.2 Data in Transit

  • HTTPS / TLS 1.3 wajib untuk semua koneksi client ↔ Firebase.
  • HSTS aktif via header Strict-Transport-Security: max-age=31536000; includeSubDomains.
  • Auth tokens — Firebase ID token disimpan di localStorage (di-protect oleh same-origin policy + CSP); tidak dapat dibaca oleh skrip pihak ketiga karena CSP script-src tidak mengizinkan sumber eksternal yang tidak whitelisted.

2. Hardening Aplikasi

  • Content-Security-Policy strict — block script/inline-eval dari sumber tidak terdaftar (lihat firebase.json).
  • X-Frame-Options: DENY — mencegah clickjacking.
  • X-Content-Type-Options: nosniff — block MIME confusion attack.
  • Referrer-Policy: strict-origin-when-cross-origin — minimalkan leak URL ke pihak ketiga.
  • Permissions-Policy — camera/microphone di-disable by default; geolocation hanya same-origin (untuk fitur lari).
  • Firebase App Check dengan reCAPTCHA v3 — block traffic non-browser ke Firestore & Functions (lihat dokumentasi setup).

3. Authorization Model

Setiap dokumen Firestore divalidasi server-side via firestore.rules:

  • Setiap collection sensitif memerlukan auth.uid matching resource.data.uid.
  • Admin allowlist by email — hanya tiga akun internal yang berhak read/write admin_config & audit_log.
  • Default deny untuk semua collection yang tidak di-whitelist.
  • Rate-limit per dokumen (1 write/sec) untuk health_logs.
  • Field validation: tipe, panjang string, range numerik.

6. Penyimpanan Token & Sesi

  • Firebase ID token (JWT, 1-jam TTL) disimpan di-memory + IndexedDB (Firebase SDK default) — bukan cookie tradisional, tapi setara httpOnly karena CSP membatasi script yang dapat membacanya.
  • Refresh token tersimpan di IndexedDB (origin-locked); tidak dapat diakses lintas-domain.
  • Logout menghapus token; “Sign out everywhere” via Firebase Auth revoke.

7. Retensi & Penghapusan

  • Data sensor: 90 hari di Firestore, lifetime di device (opt-out).
  • Session logs: lifetime — Anda dapat hapus per-sesi dari Profile.
  • Audit log admin: 1 tahun, immutable.
  • Penghapusan akun (UU PDP / GDPR Art. 17): selesai < 30 hari dari permintaan. Lihat Privacy Policy §4.

8. Vulnerability Disclosure

Menemukan bug keamanan? Kami menyambut laporan responsible disclosure. Kebijakan lengkap: docs/SECURITY.md.

  • Email: security@aervinex.id (fallback: security@aervi.id)
  • SLA: triage < 72 jam · fix critical < 14 hari · disclosure window 90 hari
  • Scope: aervinex.web.app, aervinex.firebaseapp.com, repo GitHub publik
  • Out of scope: serangan DDoS, social engineering staf, fisik device, vendor pihak ketiga (laporkan ke vendor langsung)
  • Hall of Fame: peneliti yang melaporkan valid issue akan dicantumkan (opt-in) di halaman ini.

9. Kepatuhan Standar

  • UU PDP No. 27/2022 (Indonesia) — implementasi Art. 5–10 (hak subjek data), Art. 35 (DPO), Art. 46 (notifikasi breach < 3×24 jam).
  • GDPR principles — privacy-by-default, data minimization, purpose limitation.
  • OWASP ASVS L2 — target compliance untuk web layer (in progress).
